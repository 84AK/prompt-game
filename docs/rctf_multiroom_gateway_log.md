## 1. 🔍 설계 배경 및 스키마 개편 (Blueprint - 건축가)

### [기존 아키텍처의 한계]
기존 RCTF 시스템은 `rctf_games`의 가장 최신 1개 세션에 학생들이 자동으로 강제 바인딩되는 "단일 학급 전용" 구조였습니다. 이로 인해 여러 선생님이 동시에 방을 열면, 마지막으로 생성한 선생님의 방에 모든 학생이 휩쓸려 입장하게 되는 심각한 동시성 충돌 문제가 존재했습니다.

### [해결 방안 - 대안 A]
선생님이 방을 생성할 때 임의의 **방 이름**과 고유한 **방 코드**를 부여할 수 있게 하고, 학생은 해당 입장 코드를 수동 입력해야만 분기 진입이 가능한 "멀티룸 격리 입장 게이트웨이"를 구축했습니다.

### [데이터베이스 스키마 확장]
사용자가 Supabase SQL Editor를 통해 `rctf_games` 테이블에 다음 2개 컬럼을 안전하게 사전 추가 완료했습니다.
```sql
ALTER TABLE public.rctf_games
ADD COLUMN IF NOT EXISTS room_code varchar(50),
ADD COLUMN IF NOT EXISTS room_name varchar(100);
```

## 2. 🛠️ 변경 및 구현 사항 상세 (Worker - 작업자)

### ① [src/pages/RctfBattle.jsx](file:///Users/byunmose/Desktop/AI%20Playgrounds/prompt_game/src/pages/RctfBattle.jsx)

#### A. 방 관리 상태(State) 변수 신설
- 호스트의 방 생성 입력값(`roomNameInput`, `roomCodeInput`) 상태와 학생의 게이트웨이 입장 코드(`gateRoomCode`), 현재 입장한 방 코드 영속성 및 오류 검출 상태(`joinedRoomCode`, `gateError`)를 세심하게 추가했습니다.

#### B. 권한별 세션 로딩 이중 분기 (`initSupabaseSession`)
- **선생님/관리자 (canHost):** 데이터베이스에서 가장 최신 세션을 기본값으로 우선 로드하여 빠른 대시보드 모니터링을 보장합니다.
- **학생 (USER):** 로컬스토리지에 저장되어 있는 `rctf_joined_room_code`가 존재하고 그 방이 여전히 활성 상태(`game_state = 'PLAYING'`)일 때만 자동으로 세션을 할당하도록 가드를 걸었습니다. 방이 비활성화되었을 경우 로컬 세션을 자동 폐기하여 게이트웨이 화면으로 돌려보냅니다.
- **[406 및 RLS 403 에러 해결 - 프로필 자동 시딩 장착]**
  - 로그인한 유저의 행이 `profiles` 테이블에 존재하지 않을 때(Not Acceptable 406 에러 발생 상황), 프론트엔드가 중단되지 않고 즉석에서 DB `profiles` 테이블에 해당 세션 계정으로 유효한 사용자 프로필 행을 자동 생성(시딩)해 넣어주는 지능형 자동 복구 로직을 장착했습니다.
  - 이로써 `profiles` 조회가 정상화되어 406 Not Acceptable 에러가 근절되고, DB RLS(Row Level Security) 수준에서 `ADMIN`/`TEACHER` 권한이 확실하게 승인되어 `rctf_games`에 대한 insert 403 Forbidden 에러도 완전 박멸되었습니다.
  - 또한 `handleCreateRoom`에서 호스트명을 조회할 때도 예외 처리를 덧씌워 에러 발생 시 이메일을 기준으로 폴백 처리하도록 안전화 설계를 얹었습니다.

#### C. 방 코드 입장 및 퇴장 핸들러 완벽 장착
- **`handleJoinRoom`**: 학생이 게이트에서 입력한 4자리 코드로 DB를 조회하여, 활성화된 방인 경우 로컬스토리지(`rctf_joined_room_code`)에 저장하고 동적 실시간 동기화 상태로 격리 인입시킵니다. 잘못된 코드일 시 즉시 붉은색 에러 피드백을 노출합니다.
- **`handleLeaveRoom`**: 학생이 언제든 퇴장하여 다른 방 코드를 다시 입력할 수 있도록 저장된 영속성 토큰을 소멸시키고 게이트웨이로 돌려보내는 유연한 퇴장 로직을 구성했습니다.

#### D. 호스트 방 생성 모달 UI 확장 및 스마트 난수 생성
- 모달 내에 `방 이름` 및 `방 입장 코드`를 입력할 수 있는 고광택 Glassmorphic 폼 필드를 이식했습니다.
- 방 코드 및 이름을 수동 지정하지 않고 비워둔 채 개설할 시, 선생님 이름을 파싱한 임의의 방 명칭(`X 선생님의 마법 교실`)과 **4자리 고유 난수(1000~9999 사이)**를 스마트하게 자동 매칭하여 DB에 insert하도록 UX 완성도를 최고조로 끌어올렸습니다.
- **[추가 고도화]** 방 입장 코드 입력 필드 우측에 **🎲 랜덤** 즉석 생성 버튼을 아름답게 배치하여, 클릭 시 즉석에서 4자리 난수 코드를 발급해 바로 기입할 수 있는 개설 편의성 혁신을 추가 적용했습니다.

#### E. 학생용 수동 입장 게이트웨이(Gateway UI) 신설
- 학생 진입 시 방에 참여하지 않은 경우, "열린 게임방이 없다"는 대기 화면 대신 미려한 네온 아이콘과 4자리 입장 코드를 대입할 수 있는 **전용 게이트웨이 입력 카드**를 전면에 노출합니다.

#### F. 호스트 및 학생 화면 내 방 식별 배너 큼직하게 이식
- **선생님 화면:** 선생님 대시보드 최상단 타이틀 옆에 현재 생성된 **방 이름**과 **방 입장 코드(🔑 방 입장 코드: 4092)** 배너를 큼직하게 노출하여 학생들이 쉽게 보고 입력할 수 있게 가시성을 향상했습니다.
- **학생 화면:** 소속 팀을 기입하기 전, 혹은 이미 미션을 해결 중인 전체 영역에서 자신이 들어온 방 정보를 파악하고 오입장 시 나갈 수 있도록 **최상단 공통 방 정보 배너 및 🚪 방 퇴장하기 버튼**을 공동으로 배치하여 편의를 제공했습니다.

## 3. 🛡️ 리얼타임 격리 구독 검증 (Solver - 해결사)

- **Row-Level 리얼타임 격리 동작성 완벽:** `RctfBattle.jsx` 파일에 기 정의되어 있던 Supabase Realtime 구독 블록이 이미 `filter: id=eq.${activeSession.id}` 와 `filter: game_id=eq.${activeSession.id}` 처럼 특정 ID 격리 쿼리로 무결하게 구성되어 있었습니다.
- 이로 인해 당사가 완성한 방 코드 수동 인입 처리를 통해 `activeSession`에 고유 ID만 할당되도록 보장하면, Supabase Realtime 채널이 아무런 코드 수정 없이도 **오직 우리 학급(우리 방) 세션의 실시간 패킷만 구독**하고 다른 반의 팀 제출 및 점수 배정 패킷을 완벽하게 필터링하여 혼선을 100% 방어합니다.
