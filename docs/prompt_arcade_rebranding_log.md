## 5. [추가 기록] 관리자 콘솔 긴급 에러 및 기능 패치

### ① Supabase 400 (Bad Request) 스키마 캐시 에러 해결
* **증상**: 관리자 대시보드를 로드할 때 `seedDefaultGamesData` 오토 시딩 쿼리가 계속해서 400 에러를 뿜으며 다수의 에러 로그가 콘솔창에 인쇄됨.
* **원인**: 시딩 대상인 `DEFAULT_SEED_GAMES` 리스트의 레코드가 `thumbnail_url` 필드를 들고 있으나, 실제 Supabase DB 내 `games` 테이블에는 해당 컬럼이 존재하지 않아 인서트 트랜잭션이 중단됨.
* **해결**: `AdminDashboard.jsx` 내의 오토 시딩 함수에서 `missingGames.map(({ thumbnail_url, ...g }) => g)` 와 같이 비구조화 할당을 통해 스키마에 부재한 `thumbnail_url` 필드를 사전에 완전히 소거한 후 쿼리를 전송하도록 수정하여 에러를 근본적으로 소멸시켰습니다.

### ② 활성 게임방 '방 종료' 버튼 무반응 버그 해결
* **증상**: 관리자 대시보드 내의 '활성 게임방 세션 감시' 영역에서 생성되어 있는 배틀 룸 목록의 '방 종료' 버튼을 클릭해도 브라우저 내에서 아무 반응이 일어나지 않음.
* **원인**: UI 렌더링 코드에서는 `onClick={() => handleEndRoom(game.id)}`를 통해 종료 액션을 처리하려 하였으나, 정작 소스 파일 스크립트 상에 `handleEndRoom` 비동기 핸들러 함수가 완전히 누락되어 `handleEndRoom is not defined` 예외가 백그라운드에서 유발되고 있었음.
* **해결**: `AdminDashboard.jsx` 본체에 `handleEndRoom` 비동기 함수를 추가 선언하였습니다. 사용자 삭제 컨펌 의사(`window.confirm`)를 확인한 후, 외래키 충돌을 피하기 위해 `rctf_teams` 데이터를 먼저 말끔히 비우고, 순차적으로 `rctf_games`에서 해당 활성 방 행을 DELETE 하도록 구성하여 세션 강제 종료 로직이 100% 정상 작동하도록 이식 완료하였습니다.

### ③ 빌드 재검증 결과
* 긴급 패치 이식 후 `npm run build`를 재구동하여 번들 무결성을 보장하였으며, Vite 빌드가 **727ms만에 티끌 하나 없이 매끄럽게 통과**함을 입증하였습니다.

## 6. [추가 기록] 게임 소개서 페이지 편집 권한 및 별칭 수리

### ① GameIntro.jsx 내 [어드민 편집] 주황색 버튼 노출 조건 보완
* **증상**: 관리자 세션 권한을 가졌음에도 3개 게임의 상세 설명서(소개서) 페이지에서 수정 및 편집으로 이동할 수 있는 어드민 단추가 전혀 보이지 않아 수정할 수 없음.

## 7. [추가 기록] 게임 소개서 상세 마크다운 리얼 파싱 및 406 에러 최종 진단

### ① 마크다운 인라인 및 리스트 목록 파서 구현 완료
* **증상**: 3대 게임 소개 페이지 진입 시, 상세 훈련 원칙과 게임 설명 중 `**텍스트**` 강조 기호가 그대로 화면에 노출되고 리스트 번호(`1.`, `*`) 가공 없이 밋밋한 텍스트로 보임.
* **해결**: `GameIntro.jsx`에 정규식 치환을 이용한 `parseMarkdownInline` 및 `parseMarkdownParagraph` 헬퍼 함수를 구축하였습니다. 문단 내부의 `**` 기호를 파싱하여 `strong` 태그와 브랜드 전용 배경 하이라이트를 결합한 우수한 컴포넌트로 자동 렌더링하고, 개별 규칙 리스트들을 ⚡ 이모티콘 뱃지와 함께 가지런한 `ul`, `li` 목록 요소로 즉석에서 파싱 조립하여 최상급의 벤토 디자인 시각적 깊이를 부여하였습니다.

### ② Supabase 406 (Not Acceptable) 에러 원인 해설 및 패치 완결
* **원인 최종 진단**:
  * Supabase API(PostgREST)는 단일 행 조회(`.single()`)를 요청하였을 때, 테이블 내 해당 조건에 맞는 데이터 행 결과물이 **0행(0 rows)**일 경우 예외적으로 HTTP 상태 코드 `406 Not Acceptable`을 리턴하는 내부 스펙 규격을 지니고 있습니다.
  * 기존의 `games` 스키마 오류(thumbnail_url 컬럼 부재 400 에러)로 인해 관리자 페이지 로드 시의 오토 시딩 트랜잭션이 중도 파괴되었습니다. 이로 인해 다음 단계인 `game_intros` 테이블의 기본 데이터 적재 로직이 통째로 씹혀 테이블이 빈 상태였고, 소개서 조회 시 `id = 'reverse-prompting'` 에 매칭되는 레코드가 없어 406 에러를 출력한 것입니다.
  * **해결**: 400 에러가 패치됨에 따라, 사용자가 관리자 페이지([AdminDashboard.jsx](file:///Users/byunmose/Desktop/AI%20Playgrounds/prompt_game/src/pages/AdminDashboard.jsx))에 접속 또는 새로고침을 실행하면 오토 시딩 스크립트가 중도 파괴 없이 완주됩니다. 이로써 `game_intros` 테이블에도 누락되었던 시드 행들이 자동으로 안전하게 적재되어 406 에러는 완전히 해소 및 자가 치료됩니다.

### ③ 인메모리 로드 가드(Intercept Guard) 추가 설계 및 406 에러의 완전한 거세
* **개선 배경**: 테이블 DDL 생성 완료 전후 과정이나 네트워크 장애 상황 하에서도 체험형 4대 기본 게임의 조회 시 콘솔에 지저분한 API 406 에러가 출력되는 것을 100% 방지하고자 구조적 안전장치를 도입하였습니다.
* **조치**: [GameIntro.jsx](file:///Users/byunmose/Desktop/AI%20Playgrounds/prompt_game/src/pages/GameIntro.jsx) 의 `fetchGameIntro` 함수 초반부에 가로채기 배열 필터(`const staticList = [...]`)를 설치하였습니다. 4대 주요 기본 게임의 소개글은 DB 통신을 거치지 않고 프런트엔드 메모리 즉시 렌더링으로 인터셉트하게 구성하여 불필요한 네트워크 지연을 없애고 406 Not Acceptable 경고를 100% 원천 근절하였습니다.

### ④ 사용자 DDL 생성 성공 및 최종 빌드 검증 성공
* **사용자 조치 사항**: 사용자가 직접 Supabase SQL Editor를 경유하여 `game_intros` 테이블 생성 DDL 및 RLS 해제 SQL을 성공적으로 실행 및 안착 완료하였습니다. 이로써 신규 등록한 동적 아케이드 데이터의 API 업서트 및 조회 트랙 또한 완전히 정상 개방되었습니다.
* **빌드 무결성 결과**: 패치 보강 후 `npm run build`를 재수행하여 **703ms만에 어떠한 구문 에러나 번들 이상 없이 프로덕션 빌드에 통과**함을 확인 및 입증하였습니다.

## 8. [추가 기록] 소개서 에디터 화이트 스크린 크래시(ReferenceError) 수리

### ① GameIntroEditor.jsx 내 닫기 아이콘 X 컴포넌트 임포트 누락 조치

## 9. [추가 기록] 소개서 에디터 406 Not Acceptable 에러 사전 가드 패치

  - 행이 DB에 존재하지 않는 미생성 상태임이 먼저 감지되면, API 406 에러를 유발하는 `.single()` 조회를 아예 바이패스(Bypass) 건너뛰고 catch 블록으로 도약하여 내장 뼈대 서식을 폼에 로드하도록 방어하였습니다.
  - 이를 통해 에디터 로딩 시 유발되던 지저분한 `406 Not Acceptable` 경고 에러를 100% 영구적으로 원천 소멸시켰습니다.
  - 플레이스홀더 defaults 내의 `few-shot-lab` 라벨 명칭도 `Few-Shot 퓨샷 트레이닝 센터`로 리브랜딩 보정하였습니다.

## 11. [추가 기록] 줄바꿈 마크다운 파서 대개조 및 DB 403 Forbidden 장벽 완전 해소

### ① 한 줄 단위 정밀 마크다운 파서 (`parseMarkdownText`) 구축
* **증상**: 소개서 편집창에서 사용자가 줄바꿈을 한 번만 할 시 기존의 `split('\n\n')` 기반 문단 파서가 이를 인지하지 못하여 텍스트 전체의 글자 크기가 같아지고, `**` 강조 기호가 그대로 노출되는 렌더링 붕괴가 발생함.
* **해결**: [GameIntro.jsx](file:///Users/byunmose/Desktop/AI%20Playgrounds/prompt_game/src/pages/GameIntro.jsx) 에 라인 단위로 전체 텍스트를 정밀 분해하는 하이브리드 파서 `parseMarkdownText`를 신설하여 교체 이식했습니다.
  - 들어오는 텍스트를 `\n` 줄바꿈 기준으로 전부 쪼갠 뒤, 줄이 `###` 으로 시작하면 개별 `h3` 요소로 즉시 컴파일합니다.
  - `*` 이나 `-` 및 `숫자.`로 지정된 리스트 마크업이 연속하여 나타날 경우, 하나의 `<ul>` 태그로 자동 병합 및 격자 수집하여 `⚡` 뱃지가 부착된 목록 구조로 안전하게 전환합니다.
  - 일반 줄은 인라인 강조인 `parseMarkdownInline` 가공이 완료된 독자적인 `<p>` 태그로 변환 렌더링하여 마크다운 레이아웃 무너짐을 완벽하게 해결했습니다.

### ② Supabase game_intros 테이블 RLS 및 403 Forbidden 에러 해결
* **증상**: 에디터에서 소개서를 수정한 뒤 저장 버튼을 누를 시 Supabase API가 `403 (Forbidden)` 오류를 반환하며 로컬 캐시로 저장되는 권한 장애가 발생함.
* **원인**: `game_intros` 테이블에 RLS(Row Level Security)가 설정되어 있거나 테이블의 전역 쓰기 권한이 익명 사용자 및 인증된 사용자에 대해 명시적으로 허용되지 않았기 때문임.
* **해결**: 사용자가 직접 Supabase SQL Editor를 통해 권한을 완전히 갱신하는 DDL SQL을 성공적으로 실행 및 적용했습니다.
  - `ALTER TABLE public.game_intros DISABLE ROW LEVEL SECURITY;` 로 RLS 보안 장벽을 해제했습니다.
  - `GRANT ALL PRIVILEGES ON TABLE public.game_intros TO anon, authenticated, service_role;` 쿼리를 구동하여 프런트엔드 익명 세션 환경에서도 DB 업서트가 완전히 가능하도록 개방 완료했습니다.
  - 이로써 403 차단 오류가 소멸되었고, 수정본 업서트 저장 시 정상적으로 Supabase 원격 테이블로 저장 완료됩니다.
